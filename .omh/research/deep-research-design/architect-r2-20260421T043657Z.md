Minor observations (not blockers — for the implementer's eye, not
gating approval)
----------------------------------------------------------------

1. T8's verify_cmd uses `grep -qE "1\\..*Write.*confirmed"`, which
   will match the prose ordering. Fine. Just note that the SKILL.md
   prose must literally have a numbered list ("1. Write … confirmed")
   for that grep to land, not a bulleted list. Implementer should
   honor the numbered-list form to keep verify_cmd green.

2. T9's `REPORT_CONFIRMED_RECOVERED` event is referenced by T4's
   self-heal branch but is not in T9's verify_cmd grep list (only
   `SYNTHESIS_RETRY`, `BLOCKED_RETRIES_EXHAUSTED`, `CANCELLED` are
   checked). The event IS listed in T9's Documented events bullet, so
   it'll exist; just an asymmetry in test coverage. Trivial; will be
   caught by reading the doc.

3. T12's verify_cmd does not assert a `CANCELLED` log line despite
   the cancellation acceptance bullet. Add `grep -q CANCELLED
   .omh/logs/research-*.log` if you want the test to mechanically
   confirm M1 end-to-end. Optional; the bullet still requires it
   behaviorally.

4. C1 boundary creates a parent-context-size consideration that the
   plan acknowledges ("findings are small — 5 subtopics × ~2KB"). For
   pathological cases (subagent returns a 50KB findings file because
   it dumped raw web_extract output), parent context could grow at
   synthesize/verify time. Not a v1 blocker — the researcher role's
   structured output template caps this naturally — but worth a single
   line in the role-researcher Principles ("keep SYNTHESIS concise;
   raw extracted text belongs in citations, not the body"). Consider
   for the implementer to add when writing T1.

None of the above gates approval. They're notes for the executor.
