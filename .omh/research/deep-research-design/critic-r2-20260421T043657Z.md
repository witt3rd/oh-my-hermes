3-STRIKE RETRY (T12) — REAL EXERCISE OR JUST CLAIMED?
----------------------------------------------------------------

T12 actually exercises the path, not just claims it. Acceptance reads:

  - Force the verifier to FAIL once on a synthetic citation-fabrication
    scenario; verify state.synthesis_attempts increments to 1; log shows
    SYNTHESIS_RETRY attempt=1; synthesist re-runs with prior feedback;
    second pass PASSes.
  - Then in a separate run force three consecutive FAILs; verify
    state.phase = blocked and log shows BLOCKED_RETRIES_EXHAUSTED.

That covers both arms: the success-after-one-retry path AND the
3-strike termination path. State increment, log emission, feedback
plumbing through to the synthesist's goal (which T7 acceptance also
specifies), and the terminal blocked phase are all separately observed.
This is the right shape. M2 is satisfied.

NEW ISSUES INTRODUCED BY R2 CHANGES
----------------------------------------------------------------

NONE THAT BLOCK APPROVAL. I went looking. Saying that loudly so it
doesn't read as laziness: I tried to break the C1 boundary, the C2
collapse, and the C3 propagation chain, and could not find a new
critical hole introduced by the R2 changes.

Cosmetic / low-severity observations — DO NOT block approval, list
them so the Planner can fold them into the executor task pack if
they want:

N1 (cosmetic, verify_cmd) — Several verify_cmds use `grep -q
"[omh-role:NAME]"` (e.g., T5, T7). In grep, `[...]` is a character
class, so this matches any single character from the set, not the
literal bracketed string. Use `grep -qF '[omh-role:researcher]'` or
escape. The skill prose itself is correct; only the verify command
is loose. Trivial fix.

N2 (low) — T5's W1 ordering protects state-file integrity but does
not protect against a partial batch write. If the parent writes
findings file 1 of 3 then crashes, on resume the entire batch's
delegate_task is re-issued (because completed_subtopics wasn't
updated). That re-runs 3 web-search subagents to get back to the
same point. Correct, just wasteful. Acceptable for v1; could be
improved later by per-subtopic state updates inside the batch loop.
Not worth holding the plan for.

N3 (low) — In the all-empty-results edge case (every subtopic
legitimately returns `(insufficient sources for this subtopic)`),
the synthesist will produce a near-empty report, the verifier will
PASS (per C3's explicit rule), and the user gets a confirmed but
content-free report. That's not a loop (the C3 fix prevents the
3-strike trap), but it IS a "useless artifact" outcome. Worth one
line in the README's Known Gaps: "if all subtopics return
insufficient sources, the report is confirmed but largely empty;
inspect before downstream consumption." Not a blocker; the loop
trap was the actual risk and it is closed.

N4 (informational) — T4's "refresh / view / cancel" prompt for an
existing confirmed report on the same topic depends on being able
to find that report, but the slug includes a random4 suffix, so a
naive filename match won't work. The implementer needs to scan
`.omh/research/*-report.md` frontmatter for `topic:` matching the
new request. Not a defect — just a flag for the executor to read
T4 carefully. The acceptance text is correct in intent; the
implementation detail just isn't free.

N5 (informational) — C1 inlines BOTH report and all findings into
the verifier's context. For a 5-subtopic research session this is
~15-25KB of text in the goal/context — well within budget, but
worth knowing. No action needed at v1.
